# 第 4-7 章 · 智能体范式构建 → 自建框架

## 第 4 章 · 三种经典范式

### 核心问题

第 1 章的天气 Agent 哪里不够好？

### 关键概念

**ReAct（Reasoning and Acting 推理与行动）**
- 边想边做：Thought → Action → Observation 循环
- 优势：灵活，动态调整
- 瓶颈：开销大，多步后可能迷失方向
- 适用：多步开放任务

**Plan-and-Solve（先规划后执行）**
- 先制定完整计划，再严格按计划执行
- 优势：高效，减少跑偏
- 瓶颈：一步错步步错，无法修正
- 适用：步骤清晰的结构化任务（SOP、数据流水线）

**Reflection（反思修正）**
- 执行完后自我审查，发现问题则重新执行
- 不单独使用，叠加在 ReAct 或 Plan-and-Solve 上
- 解决了"没有自我审查机制"的问题

### 工程要点

- 三种范式不是互斥的，实际产品里经常组合使用
- HelloAgentsLLM 比第 1 章的客户端多了 `function_call` 方法（支持 tools 参数）
- 最大步数限制是所有 Agent 系统的基本保险

## 第 5 章 · 低代码平台

### 核心分类

| 类型 | 代表 | 本质 |
|------|------|------|
| Workflow式 | n8n | 拖拽流程图，LLM 只是组件之一 |
| AI Native | Dify / Coze | 以 LLM 为推理核心，工具自主选择 |

### Dify vs Coze

- Dify：开源可自部署，偏企业端
- Coze：偏消费端，一键发布到抖音/飞书

### 核心区别（再谈 Workflow vs Agent）

| | Workflow（n8n） | Agent（Dify/Coze） |
|---|---|---|
| 流程定义 | 工程师写死 | 模型动态决定 |
| 工具选择 | 拖拽连接 | 模型自主选择 |
| 修改行为 | 重画流程图 | 改 prompt |

## 第 6 章 · 框架开发实践

### 三个主流框架

| 框架 | 核心模式 | 适合场景 |
|------|---------|---------|
| AutoGen | 多 Agent 对话 | 多 Agent 协作 |
| AgentScope | 分布式部署 | 高并发生产环境 |
| LangGraph | 状态图（State Graph） | 超长多步任务，精细控制 |

### LangGraph 核心概念

- **State（状态）**：Agent 当前的状态
- **Node（节点）**：一个执行步骤（调用 LLM、工具、人工输入）
- **Edge（边）**：节点间的流转条件

ReAct 的 Thought → Action → Observation = LangGraph 的 Node → Node → Node + Edge

### 框架选型依据

快速原型 → Dify；复杂流程 → LangGraph；多 Agent → AutoGen；生产部署 → AgentScope；完全定制 → 自建框架

## 第 7 章 · HelloAgents 框架设计

### 为什么自建框架

不是因为框架不好，是因为它们帮你处理了太多事，你反而不知道它帮你做了什么。

核心价值：**从"使用者"变成"设计者"**，遇到 Bug 能自己修，想加功能能自己加。

### 核心设计

**万物皆为工具（Everything is a Tool）**
- Memory、RAG、MCP 全部抽象为 Tool
- 核心逻辑只有一个：Agent 调用 Tool，Tool 返回结果
- 减少概念数 = 降低学习成本

**轻量级 + 教学友好**
- 代码按章节分开，V0.1 → V1.0 逐步迭代
- 只依赖 OpenAI SDK + 基础库

### 渐进式迭代路线

| 版本 | 章节 | 新增能力 |
|------|------|---------|
| V0.1 | 第 7 章 | Agent + Tool + Message |
| V0.2 | 第 8 章 | + Memory |
| V0.3 | 第 9 章 | + Context Engineering |
| V0.4 | 第 10 章 | + MCP |
| V1.0 | 第 11-12 章 | + Agentic RL + Evaluation |

### 工程要点

- 故障处理：工具超时时指数退避重试 + 自然语言降级
- 框架的价值不在"功能多"，在"自己能动手"
