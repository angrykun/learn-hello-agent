---

## 关键概念

### 1. 智能体的本质：感知-行动闭环

任何智能体的核心都是这个循环：

```
感知（Sensors）→ 推理（Brain）→ 行动（Actuators）→ 环境反馈
```

LLM 出现之前，软件只能"执行预设规则"。LLM 给了智能体"推理引擎"，但模型的输出只是文本 token，没有手脚。Function Calling 解决了这个问题——模型决定"做什么"，代码负责"做到"。

### 2. 传统范式演进是被问题复杂度倒逼的

| 范式 | 解决的问题 |
|------|-----------|
| 反射式 | 当前输入不足以决策 → 引入内部世界模型 |
| 基于模型 | "现在"不够用 → 引入目标导向的规划 |
| 基于目标 | 多目标冲突 → 引入效用函数权衡 |
| 学习型 | 规则不够用 → 引入强化学习自我进化 |

这不是技术偶然，是**问题驱动**的演进。

### 3. LLM 智能体的本质变化

| | 传统智能体 | LLM 智能体 |
|---|---|---|
| 知识来源 | 工程师显式编程 | 海量数据隐式预训练 |
| 能力边界 | 确定性的，有穷的 | 涌现性的，模糊的 |
| 处理输入 | 结构化数据 | 自然语言指令 |

**关键区分：LLM 的"智能"是模式识别，不是知识存储。** 它学会的是语言中的依赖关系，不是事实本身。

### 4. Function Calling 机制

**本质：决策与执行解耦**

```
模型思考 → 返回结构化指令 {tool: "get_weather", args: {...}}
          → 代码执行工具
          → 结果注入上下文
          → 模型生成最终回答
```

**requires_action 精确含义：**
- 不是"需要用户批准"
- 是"模型决定需要工具结果才能回答，代码负责执行"
- 人工确认的时机在业务代码层，不在 API 层

### 5. LLM 的三大能力边界

| 边界 | 后果 | 催生的工程方向 |
|------|------|---------------|
| 幻觉 | 信息可能错误 | RAG（真实数据源） |
| 长程推理失败 | 多步任务质量下降 | Memory（记忆系统） |
| Token 限制 | 长对话信息丢失 | 上下文工程（摘要/截断） |

---

## 工程要点

1. **智能体 = LLM（大脑）+ 工具（手脚）+ 感知（反馈）**
2. Function Calling 是架构解耦：模型擅长推理，代码擅长执行，各自做最擅长的事
3. AutoGPT 失败的三层原因：推理方向失控 + 工具调用不可靠 + 缺少自我纠正机制 → 催生 ReAct/Plan-and-Solve 等范式
4. 2023 年 Agent 爆发三要素：GPT-4 推理能力 + Function Calling 接口 + 开发者生态
